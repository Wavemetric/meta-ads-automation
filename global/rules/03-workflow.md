# 개발 방식 원칙

## 새 기능은 설계 먼저 (SDD)

코딩 전에 반드시 아래 순서를 따른다.

```
1. SPECIFY  →  뭘 만드는지 정리
               "어떤 사람이, 어떤 상황에서, 무엇을 할 수 있어야 하는가"

2. PLAN     →  어떻게 만들지 계획
               단계 나누기, 기술 선택, 각 단계 완료 기준

3. IMPLEMENT →  단계별 구현 + 확인
               한 단계 완료 확인 후 다음 단계 진행
```

"바로 만들어줘" 요청이 와도 먼저 SPECIFY 단계를 진행한다.
코딩 전에 "뭘 만드는지" 정리하면 나중에 갈아엎는 일이 줄어든다.

## 커밋 원칙

```
feat:     새 기능 추가       feat: 로그인 기능 추가
fix:      버그 수정          fix: Supabase 초기화 오류 수정
docs:     문서 수정          docs: README 설치 방법 업데이트
refactor: 구조 개선          refactor: 유저 서비스 분리
chore:    기타 설정 변경     chore: .gitignore 업데이트
```

기능이 완성되거나 중요한 변경이 있을 때마다 커밋한다.
"나중에 한꺼번에 커밋"은 하지 않는다.

## 파일 저장 위치

| 종류 | 위치 |
|------|------|
| 원본 데이터 | `data/input/` |
| 결과물/출력물 | `data/output/` |
| 스크립트 | `scripts/` |
| 데이터 파일(csv, xlsx 등) | Git 추적 안 함, 로컬에서만 관리 |

---

## 서버 이관 기준

로컬에서 정상 동작 확인 후:
- **주 1회 이상 반복 실행** 필요 → 서버 이관 요청 (GitHub Actions 등)
- **그 이하** → 로컬 자동 실행으로 충분

---

## Claude Code 사용 원칙

- 파일 수정 전 반드시 먼저 읽는다
- 모르는 것은 추측하지 않고 확인한다
- 한 번에 너무 많은 것을 바꾸지 않는다 (단계별로)
- 기존 코드 구조를 무너뜨리지 않는다 (기능 추가 시)
